# Stage 0: Планування стеку проекту Ensemble_AI

## Коротко про проект

**Ensemble_AI** — це модульна платформа для оркестрації ансамблю локальних великих мовних моделей (LLM).

**Суть роботи:**

- Клієнт надсилає запит.
- Запит паралельно розсилається на кілька незалежних **worker-моделей** (Qwen/Gwen, що працюють локально).
- Відповіді від workers надсилаються до **Арбітра** (більша модель).
- Арбітр перевіряє коректність відповідей, виправляє помилки (без генерації всього з нуля) і формує оптимальну фінальну відповідь.
- Клієнт бачить **тільки фінальну відповідь**.

**Ключові вимоги:**
- Усі моделі працюють локально (без зовнішніх API).
- Обмежень по вазі моделей немає.
- Платформа модульна — легко підключати/змінювати worker-моделі та Арбітра.
- Worker-моделі передають дані Арбітру (JSON або текст).
- Арбітр зчитує, обробляє та повертає відповідь користувачу.
- Стек: Python + Docker + БД.

## Фінальний стек

### Frontend
- **Next.js 15** (App Router)
- Tailwind CSS + shadcn/ui

### Backend
- **Python 3.12+**
- **FastAPI** — Orchestrator
- **Celery + Redis** — черги та паралельна обробка
- **PostgreSQL** — база даних

### LLM частина
- **Workers**: окремі Docker-контейнери на базі Ollama
- **Arbiter**: окремий Python-сервіс
- **Inference**: Ollama (основний)

## Повне планування стеку

### 1. Основні технології
- Мова: Python 3.12+
- Контейнеризація: Docker + Docker Compose
- База даних: PostgreSQL (моделі, system prompts, configs, логи, історія)
- Web/API: FastAPI
- Асинхронність: Celery + Redis
- Оркестрація: custom Python-модулі

### 2. Архітектура системи
- **Client**: Next.js
- **Orchestrator** (FastAPI): приймає запит, розсилає задачі в чергу
- **Workers**: незалежні Docker-контейнери (працюють паралельно)
- **Arbiter**: окремий сервіс, який отримує відповіді від workers, аналізує їх і формує фінальну відповідь
- **DB**: зберігання конфігурацій, промптів та історії

### 3. Модульність
- Конфігурація моделей через YAML + БД
- Абстрактний клас `BaseWorker`
- Arbiter як окремий модуль
- Легке додавання нових моделей через конфіг

### 4. Наступні етапи
- **Stage 1**: Docker Compose + PostgreSQL + FastAPI skeleton
- **Stage 2**: Інтеграція Ollama + базові worker-моделі
- **Stage 3**: Pipeline worker → arbiter
- **Stage 4**: Повна логіка Арбітра
- **Stage 5**: Next.js Frontend + UI + фінальне тестування
